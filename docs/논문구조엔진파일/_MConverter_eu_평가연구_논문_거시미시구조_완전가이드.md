<td><p><strong>서론: 평가 연구 논문의 구조적 원리</strong></p>
<p>평가 연구(Evaluation Research) 논문은 프로그램·정책·조직·시스템의 가치(merit), 유용성(worth), 의의(significance)를 체계적으로 판단하는 고유한 학술적 구조를 갖는다. <p><strong>▌거시적 구조 (Macro Structure):</strong> Stufflebeam의 CIPP 모형과 평가 표준에 기반한 각 장 간 연결 논리</p>
<p>→ 평가 맥락(Context) → 투입 평가(Input) → 과정 평가(Process) → 산출 평가(Product)</p>
<p>→ 각 장의 선택이 평가의 목적(형성적/총괄적)·접근법·가치 기준과 일관성을 유지해야 한다</p>
<p><strong>▌미시적 구조 (Micro Structure):</strong> Patton의 활용 중심 평가에 기반한 각 장 내부 문단·문장 수준의 배열 원리</p>
<p>→ Patton(1997): '의도된 사용자(intended users) → 의도된 사용(intended uses) → 평가 질문 → 방법 선택 → 결과 보고'의 활용 논리 사슬</p>
<p>→ 평가 결과가 실제 의사결정에 활용되도록 이해관계자 참여와 보고 전략이 문장 수준에서 구현되어야 한다</p>
<p><strong>핵심 차이:</strong> 평가 연구 논문은 '사실 발견'이 아닌 '가치 판단(value judgment)'을 핵심으로 하며, 연구자가 평가자(evaluator)로서의 가치 기준과 판단 근거를 명시적으로 드러내야 한다.</p></td>

| **제1부: 거시적 구조 --- Stufflebeam의 CIPP 모형과 평가 표준 기반 각 장 간 연결 논리** |

Stufflebeam & Coryn(2014)은 프로그램 평가의 가장 포괄적 모형으로 CIPP(Context-Input-Process-Product) 모형을 제시한다. 이 모형은 평가 연구 논문의 거시적 구조---즉 각 장이 어떤 역할을 수행하고 장과 장이 어떤 논리로 연결되는가---의 핵심 골격이다. 동시에 Joint Committee의 프로그램 평가 표준(유용성·실행가능성·적절성·정확성·평가 책무성)이 각 장에서 어떻게 구현되는지를 명시해야 한다.

|  |  |  |  |
|:--:|:--:|:--:|:--:|
| **장** | **핵심 질문** | **핵심 산출물** | **다음 장으로의 연결** |
| 1\. 서론/평가 맥락 | 왜 이 평가인가? 무엇을 평가하는가? | 평가 대상 + 평가 목적 + 평가 질문 | 맥락 → 이론적 배경 방향 |
| 2\. 이론적 배경/평가 모형 | 어떤 이론·모형으로 평가하는가? | 평가 모형 + 분석 틀 + 가치 기준 | 모형 → 평가 설계 근거 |
| 3\. 문헌연구/선행 평가 검토 | 기존 평가는 어떠했는가? | 선행 평가 지형도 + 평가 공백 | 공백 → 본 평가 정당화 |
| 4\. 평가 설계/방법론 | 어떻게 평가하는가? | 평가 설계 + 자료 수집 + 분석 방법 | 설계 → 결과 도출 |
| 5\. 평가 결과 | 평가 결과는 무엇인가? | 평가 기준별 판정 + 증거 제시 | 결과 → 가치 판단 |
| 6\. 논의/가치 판단 | 이 결과는 무엇을 의미하는가? | 가치 판단 + 이론 기여 + 개선 권고 | 판단 → 권고·활용 |
| 7\. 결론/권고·활용 전략 | 무엇을 권고하는가? | 개선 권고 + 활용 전략 + 메타평가 | 수미상관: 평가 목적 답변 |

**제1장. 서론 / 평가 맥락 (Evaluation Context)**

**1.1 거시적 기능**

평가 연구 논문의 서론은 Stufflebeam의 CIPP 모형 중 \'맥락 평가(Context Evaluation)\'에 대응한다. 이 장은 평가 대상(evaluand)의 성격과 맥락을 명확히 규정하고, 왜 이 평가가 필요한가를 논증하며, 평가의 목적·범위·핵심 질문을 선언하는 기능을 수행한다. Patton(1997)은 모든 평가는 의도된 사용자(intended users)와 의도된 사용(intended uses)의 명확한 식별에서 출발해야 한다고 강조한다.

<td><p><strong>평가 연구 논문 서론의 6단계 거시 기능 (Stufflebeam, 2014 + Patton, 1997 기반)</strong></p>
<p>① 평가 대상(Evaluand) 기술: 프로그램·정책·조직·시스템의 성격, 목표, 구성 요소, 대상 집단</p>
<p>② 평가 필요성(Need for Evaluation) 논증: 왜 이 시점에 이 평가가 필요한가—책무성, 개선, 지식 생산</p>
<p>③ 이해관계자 식별: Patton의 핵심—의도된 사용자(intended users)는 누구인가</p>
<p>④ 평가 목적 선언: 형성적 평가(formative) vs 총괄적 평가(summative) vs 발전적 평가(developmental)</p>
<p>⑤ 평가 질문(Evaluation Questions) 설정: 평가가 답해야 할 핵심 질문 2~5개</p>
<p>⑥ 평가 범위·한계 명시: 시간적·공간적·개념적 범위와 의도적 배제 영역</p>
<p><em>출처: Stufflebeam &amp; Coryn (2014), Ch. 1, 13; Patton (1997), Ch. 1-3</em></p></td>

<p><strong>전제 4 — 평가 질문의 실용성:</strong> 평가 질문은 학문적 호기심이 아닌 의사결정 필요에 의해 도출되어야 한다.</p>

**제2장. 이론적 배경 / 평가 모형 (Evaluation Models & Theoretical Framework)**

**2.1 거시적 기능**

평가 연구의 이론적 배경은 순수 학술 연구와 달리, 평가 대상을 어떤 모형·기준·가치 틀로 평가할 것인가를 정당화하는 \'평가 프레임워크 구축\'에 초점을 맞춘다. Stufflebeam & Coryn(2014)은 26가지 평가 접근법을 체계적으로 분류하고, 각 접근법의 강점·한계를 Joint Committee의 프로그램 평가 표준에 따라 평가한다. 이론적 배경 장에서 연구자는 이 분류 체계 속에서 자신의 평가 접근법을 위치시키고 정당화해야 한다.

<p>Stage 1: 평가 개념 문제 제기 — '평가(evaluation)'의 다의성과 접근법 간 논쟁 지형 확인</p>
<p>Stage 2: 핵심 평가 개념 정의 — 가치(merit), 유용성(worth), 의의(significance)의 조작적 정의 확정</p>

<p>Stage 6: 평가 프레임워크 통합 구성 — 복수 모형·이론을 통합하여 본 연구의 평가 분석틀 구축</p>
<p>Stage 7: 평가 기준(Evaluation Criteria) 도출 — 분석틀에서 프로그램을 판단할 구체적 평가 기준 체계 도출</p></td>

**2.3 Joint Committee 프로그램 평가 표준 (평가의 질 판단 기준)**

|  |  |  |
|:--:|:--:|:--:|
| **표준 범주** | **핵심 질문** | **평가 연구 논문에서의 위치** |
| 유용성(Utility) | 평가 결과가 이해관계자에게 유용한가? | 서론(의도된 사용자·사용) + 결론(활용 전략) |

| 평가 책무성
(Evaluation Accountability) | 평가 자체가 적절히 수행되었는가? | 결론(7장) --- 메타평가(metaevaluation) |

**제3장. 문헌연구 / 선행 평가 검토 (Literature Review & Prior Evaluations)**

**3.1 거시적 기능**

평가 연구의 문헌연구는 두 가지 독자적 기능을 수행한다. 첫째, 평가 대상 프로그램·정책에 대한 기존 연구와 이론적 근거를 검토한다(프로그램 이론 검토). 둘째, 이 프로그램·정책에 대한 기존 평가 연구의 성과와 한계를 체계적으로 분석한다(선행 평가 검토). Stufflebeam & Coryn(2014)은 평가자가 프로그램의 이론적 기반(program theory)과 기존 평가 증거를 모두 검토할 것을 요구한다.

**제4장. 평가 설계 / 방법론 (Evaluation Design & Methodology)**

**4.1 거시적 기능**

평가 설계 장은 평가 연구의 방법론적 핵심으로, 무엇을 어떻게 평가했는가를 넘어 왜 이 평가 설계인가에 대한 정당화가 필수적이다. Patton(1997)은 평가 방법은 평가 목적과 의도된 사용에 의해 결정되어야 한다고 강조하며, 방법론 교조주의(methodological orthodoxy)를 경계한다. Stufflebeam & Coryn(2014)은 평가 설계가 Joint Committee 표준의 정확성(Accuracy) 기준을 충족해야 한다고 요구한다.

<p><strong>원리 1 — 방법론적 적절성(Methodological Appropriateness):</strong> 방법은 평가 질문의 성격에 의해 결정된다. '효과가 있었는가'는 양적 방법을, '어떻게 경험되었는가'는 질적 방법을, '얼마나 효율적인가'는 비용-효과 분석을 요구한다.</p>

**제5장. 평가 결과 (Evaluation Findings)**

**5.1 거시적 기능**

평가 결과 장은 평가 질문에 대한 체계적 답변을 제시하는 장이다. 일반 연구 논문의 결과 장과 달리, 평가 결과는 사실 제시(findings)와 가치 판단(judgments)을 구분하되 연결해야 한다. Stufflebeam의 CIPP 모형에 따르면, 결과는 맥락·투입·과정·산출의 4차원에서 체계적으로 제시되어야 한다. Patton(1997)은 결과 보고가 의도된 사용자의 이해 수준과 의사결정 필요에 맞게 구성되어야 한다고 강조한다.

<col style="width: 96%" />

<tbody>
<tr>
<td><p><strong>CIPP 모형에 따른 평가 결과의 4차원 구조</strong></p>
<p><strong>차원 1 — 맥락 평가 결과 (Context):</strong> 프로그램이 대응하는 필요(needs)가 실제로 존재하는가? 대상 집단의 필요가 적절히 진단되었는가?</p>
<p><strong>차원 2 — 투입 평가 결과 (Input):</strong> 프로그램의 전략·설계·자원 배분이 적절한가? 대안적 접근과 비교하여 이 설계가 최선인가?</p>
<p><strong>차원 3 — 과정 평가 결과 (Process):</strong> 프로그램이 계획대로 실행되었는가? 실행 충실도(fidelity)는 어떠한가? 예상치 못한 문제가 발생했는가?</p>
<p><strong>차원 4 — 산출 평가 결과 (Product):</strong> 프로그램이 의도된 성과를 달성했는가? 의도하지 않은 결과는 무엇인가? 장기적 영향과 지속가능성은?</p>
<p><em>출처: Stufflebeam &amp; Coryn (2014), Ch. 13</em></p></td>

| 1\. 분석 개요 | 분석 목적·데이터·방법 요약 | 평가 질문과 분석 방법의 대응 구조 재확인 |
| 2\. 맥락·투입 평가 결과 | 프로그램 맥락의 적절성, 설계의 타당성 | 필요 진단 결과 + 프로그램 논리 모형(Logic Model) 검증 |
| 3\. 과정 평가 결과 | 실행 충실도, 참여율, 운영 현황 | 계획 대비 실제 실행의 편차 분석 |

| 6\. 결과 요약 | 핵심 발견 통합 정리 | 평가 질문별 답변 요약 + 논의 장 예고 |

**제6장. 논의 / 가치 판단 (Discussion & Value Judgments)**

**6.1 거시적 기능**

평가 연구의 논의 장은 일반 연구 논문과 구별되는 고유한 기능을 수행한다. 일반 논문의 논의가 \'이 결과가 이론에 무엇을 의미하는가\'를 묻는다면, 평가 논문의 논의는 \'이 결과에 기초하여 프로그램의 가치를 어떻게 판단하는가\'를 묻는다. Stufflebeam은 평가의 본질이 가치 판단(valuing)임을 반복 강조하며, 논의 장에서 이 판단이 명시적으로 이루어져야 한다고 요구한다. Patton(1997)은 평가 결과의 해석이 이해관계자와의 협력적 의미 구성(collaborative meaning-making) 과정을 거쳐야 활용 가능성이 높아진다고 주장한다.

**제7장. 결론 / 권고·활용 전략 (Conclusion, Recommendations & Utilization)**

**7.1 거시적 기능**

평가 연구 결론의 고유한 특성은 단순한 요약을 넘어 구체적 개선 권고(recommendations)와 평가 활용 전략(utilization strategy)을 포함해야 한다는 점이다. Patton(1997)은 결론의 핵심이 \'평가 결과가 실제로 사용되도록 하는 전략\'에 있다고 강조하며, 이를 위해 의도된 사용자와의 상호작용적 결과 보고(interactive reporting) 전략을 제안한다. Stufflebeam은 결론에 메타평가(metaevaluation)---즉 이 평가 자체의 질을 평가 표준에 따라 자가 점검하는 과정---를 포함할 것을 권고한다.

<p><strong>유형 1 — 필요 사정 질문 (Needs Assessment):</strong> 이 프로그램이 대응하는 필요는 실제로 존재하는가? 얼마나 심각한가?</p>

<p><strong>유형 6 — 지속가능성 질문 (Sustainability):</strong> 프로그램 종료 후에도 효과가 지속될 수 있는가?</p>
<p><strong>유형 7 — 의도하지 않은 결과 질문 (Unintended Consequences):</strong> 예상하지 못한 긍정적/부정적 결과는 무엇인가?</p>

| 3 | 의도된 사용자 명시 | 본 평가의 의도된 사용자(primary intended users)는 \[사용자 목록\]이며, 이들은 평가 결과를 \[의사결정 유형\]에 활용할 것으로 기대된다(Patton, 1997). |
| 4 | 평가 목적 선언 | 본 평가는 \[프로그램\]의 \[측면\]을 \[형성적/총괄적\]으로 평가하여 \[목적\]에 기여하는 것을 목적으로 한다. |
| 5 | 평가 질문 제시 | 구체적으로, 본 평가는 다음의 질문에 답한다: (1) \[실행 충실도 질문\] (2) \[효과성 질문\] (3) \[효율성 질문\] |
| 6 | CIPP 프레임 적용 선언 | 본 평가는 Stufflebeam(2014)의 CIPP 모형에 기반하여 프로그램의 맥락·투입·과정·산출을 포괄적으로 사정한다. |
| 7 | 활용 중심 접근 선언 | Patton(1997)의 활용 중심 평가 원리에 따라, 본 평가는 의도된 사용자의 정보 필요에 부응하는 평가 질문과 방법을 설계하였다. |
| 8 | 프로그램 논리 모형 예고 | 평가에 앞서 프로그램의 논리 모형(Logic Model)을 구성하여 투입-활동-산출-성과의 인과 경로를 명시하고, 이를 평가의 분석틀로 활용한다. |
| 9 | 이해관계자 갈등 구조 | 이 프로그램에 대한 이해관계자들의 관점은 상이하며, \[집단A\]는 \[기대\], \[집단B\]는 \[우려\]를 표명하고 있어 다관점 평가가 요구된다. |
| 10 | 형성적 목적 강조 | 본 평가의 주된 목적은 프로그램의 지속 여부를 판단하는 것이 아니라, 실행 과정에서 발견된 문제를 진단하고 개선 방안을 도출하는 형성적 기능에 있다. |
| 11 | 책무성 근거 | \[기관\]의 \[법률/규정\]에 따라 이 프로그램은 \[주기\]마다 체계적 평가를 수행할 의무가 있으며, 본 평가는 이 제도적 요구에 대응한다. |
| 12 | 평가 범위 한정 | 본 평가는 \[프로그램\]의 \[특정 구성요소\]에 한정하며, \[관련 영역\]은 분석 범위에서 제외한다. |

**3. 서론 탈락 패턴 10가지**

|  |  |  |
|:--:|:--:|:--:|
| **번호** | **탈락 패턴** | **문제점 및 개선 방향** |
| 1 | 의도된 사용자 미명시 | 누가 이 평가를 사용할지 불명확. → Patton의 원리에 따라 의도된 사용자·사용 명시 필수 |
| 2 | 평가 질문 부재/모호 | 무엇을 평가하는지 불명확. → 구체적 평가 질문 2\~5개 제시 필수 |
| 3 | 평가 목적 유형 미선언 | 형성적인지 총괄적인지 불명확. → 평가 목적 유형 명시적 선언 |
| 4 | 프로그램 기술 불충분 | 평가 대상의 성격이 불명확. → 프로그램 목표·대상·활동·자원 체계적 기술 |
| 5 | 평가 필요성 논증 부재 | 왜 이 평가가 필요한지 정당화 없음. → 책무성/개선/지식 생산 동기 논증 |
| 6 | 이해관계자 분석 누락 | 관련 행위자 식별 없음. → 주요 이해관계자 목록과 관점 제시 |
| 7 | 연구 논문과 구분 불명 | 평가가 아닌 일반 연구처럼 서술. → 가치 판단·활용 목적 명시 |
| 8 | 평가 모형 미언급 | 어떤 평가 접근법인지 불명확. → 채택 모형 명시(CIPP, 활용 중심 등) |
| 9 | 논리 모형(Logic Model) 부재 | 프로그램의 인과 논리 미제시. → 투입-활동-산출-성과 논리 모형 구성 |
| 10 | 평가 범위 과대/과소 | 너무 넓거나 좁은 범위. → 평가 가능한 범위로 적절히 한정 |

| S4: 프로그램 현황·성과 개관문 | 프로그램의 현재 운영 현황과 알려진 성과를 간략히 개관 | \[기간\] 동안 총 \[N\]명이 참여하였으며, \[알려진 성과\]가 보고되고 있다 |
| S5: 평가 필요성 예고 전환문 | 이 프로그램에 대한 체계적 평가 필요성을 예고하고 P2로 전환 | 그러나 이 프로그램의 \[측면\]에 대한 체계적 평가는 아직 수행되지 않아 근거 기반 의사결정이 어려운 상황이다 |

| S1: 평가 질문 재진술문 | 해당 평가 질문을 재진술하고 분석 방법 요약 | \[평가 질문 N\]: \[질문 내용\]에 답하기 위해 \[분석 방법\]을 적용하였다 |

| S3: 질적 증거 제시문 | 인용·사례·관찰 자료를 보완적으로 제시 | 이해관계자 면담에서 \[참여자\]는 \'\[인용\]\'이라고 진술하여 이 수치적 결과를 뒷받침한다 |
| S4: 기준 대비 판정문 | 성과 기준 대비 달성 여부를 명시적으로 판정 | 이상의 결과에 기초하여, \[프로그램\]의 \[측면\]은 \[기준\]에 \[도달/미달/부분 도달\]한 것으로 판정된다 |
| S5: 예외·부정적 결과 보고문 | 의도하지 않은 결과, 예외 사례, 부정적 발견도 투명하게 보고 | 다만, \[하위 집단/조건\]에서는 \[부정적/예외적 결과\]가 관찰되었으며, 이는 \[의미\]를 시사한다 |
| S6: 다음 질문/논의 전환문 | 다음 평가 질문으로 이행하거나 논의 장과 연결 | 이 결과의 이론적·실천적 함의는 제6장에서 상세히 논의된다 |

<td><p><strong>가치 판단 문단의 3층 미시 구조</strong></p>
<p><strong>Layer 1 — 증거 기반 가치 판단 (Evidence-Based Valuing):</strong></p>
<p>→ 평가 결과(5장)에 기초하여 프로그램의 가치(merit)·유용성(worth)·의의(significance)를 판정</p>
<p>→ 문장 형식: '평가 결과를 종합하면, [프로그램]의 [측면]은 [높은/중간/낮은] 수준의 가치를 갖는 것으로 판단된다'</p>
<p><strong>Layer 2 — 가치 판단의 근거 논증 (Justification of Judgment):</strong></p>
<p>→ 왜 이런 판단에 도달했는지를 평가 기준·증거·논리로 논증</p>
<p>→ 문장 형식: '이 판단의 근거는 [기준A]에서 [결과], [기준B]에서 [결과]의 종합적 고려에 있다'</p>
<p><strong>Layer 3 — 판단의 실천적 함의 (Practical Implications of Judgment):</strong></p>
<p>→ 이 가치 판단이 의도된 사용자의 의사결정에 갖는 구체적 함의 도출</p>
<p>→ 문장 형식: '이 판단은 [프로그램 개선/지속/확대/축소]의 방향에서 [구체적 함의]를 시사한다'</p>

| 1 | 서론 반복형 요약 | 서론 문장 그대로 반복 → 평가 완료 후 메타적 재서술 |
| 2 | 권고 부재/모호 | 구체적 권고 없이 \'개선이 필요하다\'만 → 평가 결과 직접 근거 권고 |
| 3 | 활용 전략 미제시 | 결과 보고만, 활용 방안 없음 → Patton 원리 기반 사용자별 활용 전략 |
| 4 | 메타평가 부재 | 이 평가 자체의 질 점검 없음 → Joint Committee 표준 대비 자가 점검 |
| 5 | 가치 판단 회피 | 판단 없이 \'추가 연구 필요\'만 → 가용 증거 기반 명시적 판단 |
| 6 | 과도한 일반화 | 단일 평가 결과를 과대 일반화 → 평가 범위 내 결론 한정 |
| 7 | 한계의 형식적 나열 | 한계를 한 줄로 → 한계의 영향과 보완 방법 상술 |
| 8 | 이해관계자 무시 | 사용자별 차별적 메시지 없음 → 의도된 사용자별 맞춤 메시지 |
| 9 | 실행 방안 부재 | 권고만, 실행 방법 없음 → 단계·담당·시간표 포함 실행 방안 |
| 10 | 마지막 문장 약함 | 끝이 흐지부지 → 평가 메시지 각인하는 강한 마지막 문장 |
